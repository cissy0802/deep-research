# Deep Research 工作流程(engine 规程)

本仓库产出对抗验证过的深度研究文章(plain/deep × zh/en 四版)。任何一期研究都必须走完下述流程;做不到的环节必须显式降级标注,不允许静默跳过。

## 目录约定

```
TOPICS.md                    选题清单:已发布 / 待研究 / 候选池;流程头是本规程的摘要
research/<slug>/             本期研究底稿(发布不依赖它,但保留作验证底座存档)
  STATE.md                   断点恢复文件:计划、调研线、运行日志、「下一步」恢复点
  01-raw-claims.md           Round 1 原始论断(按调研线组织,标注来源类型)
  02-verified-claims.md      承重论断 + 验证结论 + 证据分级
  03-verification-round2.md  3 票对抗验证判决(如有)
  04-verification-round3.md  反证搜索席 + 方法学审计席判决(如有)
sources/<slug>-{plain,deep}.{zh,en}.md   发布源文件(四版)
build.py                     渲染引擎:ARTICLES/TLDRS/CHIPS/INDEX_ENTRIES(+可选 FIGURES)注册
```

## 流程(六步)

### 1. 选题登记
在 TOPICS.md「待研究」区新增条目(编号递增,格式:一句话/为什么适合/理论底座/关键争议/实证锚点/注意),单独 commit。选题标准:流行叙事强、争议大、但存在可核查的一手证据——对抗验证增值最大的题目。

### 2. Round 1 · 多线调研
- 拆 5-8 条调研线(写进 STATE.md),覆盖:正方一手、反方一手、历史前科/考古、厂商口径、独立测量、监管/官方数据。
- 逐线收集一手来源,产出 `01-raw-claims.md`:每条论断标注来源类型(【官方】/【研究】/【厂商】/【媒体】),文末列「交叉口径问题」清单供验证阶段用。
- 来源广度基准:承重文章 ≥30 源,标准运行 80-160 源。

### 3. Round 2 · 对抗验证(3 票制)
- 从 01 提取承重论断分组(通常 15-40 组)。
- **每组 3 票独立核验,refute-by-default**:验证者的任务是推翻,不是确认;逐字回到一手文件核对口径(分子、分母、时间窗、限定语)。
- 用子代理跑:3 个独立验证 agent 各自对全部组投票,互不见对方结论;或走云端 workflow。
- 判决记录到 `03-verification-round2.md`:HOLDS / 修正(记录修正前后口径)/ 推翻。
- 「0 推翻」本身是需要警惕的结果——见 Round 3。

### 4. Round 3 · 单源实证双席审计(#14 起强制)
对**单源承重实证**(只有一个来源支撑、且文章结论压在上面的数字)增设:
- **反证搜索席**:主动搜索独立团队+独立数据的矛盾/证实测量,记录全部搜索角度(包括没搜到的)。
- **方法学审计席**:敌意审稿(样本量、推断统计、选样程序、利益冲突),**有否决权**——被否决的数字不得承重,只能降级为弱结论或弃用。
- 判决记录到 `04-verification-round3.md`。

### 5. 证据分级与成文
分级词表(逐条标注,不可省):**多源证实**(≥2 独立来源同向)/ **单源已核**(一手可回溯)/ **方向存争**(独立来源矛盾)/ **厂商口径**(利益相关方自述,方向可参考、程度不承重)/ **现场核验** / **未验证**。转述保真 ≠ 事实为真,分级即为此而设。

四版格式(对齐 `sources/ai-capex-1999-*`):
- **deep 版**:H1 带「(深入版)/(Deep Dive)」;blockquote 方法学前言(票数/判决统计、修正记录、时效声明);`## N.` 编号章节;结尾「N 个可检验主张」(按证据强度排序,每条给检验方式)+「附:主要来源」+ 站内相关研究链接。
- **plain 版**:blockquote 指向深入版;无 inline 验证标注;结尾「最要紧的几件事」。
- zh 先写,en 忠实翻译(不是转写)。

### 6. 注册、构建、归档
1. build.py 注册四处:ARTICLES(4 tuple)、TLDRS(4 key)、CHIPS(4 key)、INDEX_ENTRIES(1 条,追加到列表末尾);`python3 build.py` 构建验证。
2. TOPICS.md 该期条目移入「已发布」:页面、底座统计(来源数/票数/判决)、核心口径记录、钩子去向。
3. **钩子收割**:提炼 ≤5 个超出本篇范围的支线问题——适合 deep research 的进「候选池」;适合某 Second Brain 站的按该站 TOPICS 格式追加到其 backlog(单独 commit,行尾标「钩子·源自 #N」)。判据是「够格才留」,非配额。
4. 分批 commit(登记/底稿/成文/注册构建各自独立),用户确认后 push。

## 降级运行规则(#15 教训)

云端 workflow 或子代理不可用、只能单线运行时:
- 方法学前言**必须**披露降级(「本期为单线核验运行,非 3 票对抗制」),分级从严:凡未回溯到一手文件的数字一律降级或弃用。
- TOPICS.md 条目和 STATE.md 标注「**对抗验证待补**」;补验前不视为标准发布。
- 降级是例外不是常态;恢复条件具备时优先补验最近一期。

## 红线

- 不允许单点引用带区间的预测(如 IEA 945 TWh 必须带情景区间)。
- 不允许把厂商口径当独立证据;「谁在说话、谁受益」必须随数字出现。
- 不允许混引不同口径(合约价 vs 现货价、承诺额 vs 落地额、申请量 vs 建成量)。
- 引语必须逐字核对;找不到一手的「名人名言」按伪引用处理。
- 已发布文章的数字修正,必须同步更新四版 + TLDRS/CHIPS + 重新构建。
